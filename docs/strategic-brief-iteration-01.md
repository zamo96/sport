# TennisSearch Strategic Brief

## Iteration Goal

Show the first strong working version of TennisSearch on Web and iOS, where a user can quickly find a partner or organize a game and move toward a confirmed meeting.

## Primary Product Priority

first confirmed game

## Success Metric

- more accepted game requests
- more confirmed games

## Guardrails

- do not break onboarding and profile setup
- keep Web and iOS functionally aligned on the core flow
- stability is more important than adding extra scope
- avoid unnecessary product complexity and overloaded screens
- do not introduce risky architectural or schema changes unless they clearly improve the core flow

## Product Principles For This Iteration

- every core screen should move the user toward a real game
- speed and clarity are more important than feature depth
- recommendations and next steps should remain understandable to the user

## Strategic Context

The team should proactively identify and improve friction in the main user journey:

- onboarding to discover
- discover to action
- search or proposal creation
- response handling
- chat to confirmed game

The team should also actively look for stability issues in the same flow, especially where state can become unclear, stale, or inconsistent between Web and iOS.

## Explicit Non-Goals

- no premium expansion in this iteration
- no large-scale redesign of the whole product
- no broad expansion into non-core secondary features before the core flow is stronger

## Output Expected From The Team

The team should return:

- the main friction points they found in the current product
- the most valuable improvements they selected for this iteration
- what was improved on Web
- what was improved on iOS
- what was improved in backend, matching, or reliability if relevant
- what product effect is expected
- what risks or weak spots remain

## Product Owner Intent

I want the team to operate with high autonomy.

The team is expected to:

- independently identify opportunities to improve the interface, product logic, and system reliability
- prioritize improvements that strengthen the path to the first real game
- keep the product simple and focused
- bring back only the decisions that truly require Product Owner input

The Product Owner will provide strategy, priorities, and tradeoff direction, not detailed implementation requirements.
