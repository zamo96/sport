# Сценарии Для Дизайн-Ревью

Этот документ задает обязательный visual review loop для команды.

Правило:

- команда смотрит не отдельные экраны в вакууме, а проходит основные product scenarios end-to-end
- по каждому сценарию фиксируются ключевые шаги интерфейса
- для iOS visual review используются screenshots из Simulator
- для Web visual review используются реальные экраны приложения, а не только чтение JSX/SwiftUI-кода
- по каждому шагу сценария должен быть screenshot, если технически возможно получить экран

## Сценарий 1. Первый Вход И Onboarding

Цель:

- понять, достаточно ли ясно пользователь проходит от первого входа до готового профиля

Шаги:

1. вход в приложение
2. старт onboarding
3. базовый профиль
4. выбор спорта и уровня
5. доступность и район
6. завершение onboarding
7. первый переход в discover

Что оценивать:

- контраст и читаемость
- перегруженность шага
- что обязательно, а что можно пропустить
- есть ли один понятный next step

## Сценарий 2. Discover И Player Card

Цель:

- понять, достаточно ли player card помогает принять решение

Шаги:

1. открытие discover
2. первая карточка игрока
3. просмотр explainability reasons
4. просмотр location / format / availability
5. лайк или пропуск
6. состояние после мэтча, если он возникает

Что оценивать:

- иерархия информации на карточке
- понятность CTA
- понятность причин “почему этот игрок”
- корректность label для `места` и `формата`
- визуальная читаемость на темных и светлых поверхностях

## Сценарий 3. Создание Поиска Игры

Цель:

- проверить, что форма меняется в зависимости от спорта и не навязывает теннисную модель всем видам спорта

Шаги:

1. открытие create game search
2. выбор спорта
3. изменение доступных полей после выбора спорта
4. выбор места
5. выбор формата
6. выбор количества участников
7. сохранение поиска

Что оценивать:

- sport-aware параметры
- корректность `court / venue / location`
- корректность format label
- понятность players-needed logic
- есть ли в форме теннисные артефакты для boxing / fitness / yoga / football / volleyball

## Сценарий 4. Прямое Предложение Игры

Цель:

- проверить сценарий proposal к существующему партнеру

Шаги:

1. открытие create game proposal
2. выбор спорта
3. выбор места
4. выбор даты и времени
5. выбор формата
6. отправка предложения
7. состояние после создания

Что оценивать:

- понятность разницы между proposal и open search
- корректность sport-aware labels
- понятность статуса после отправки

## Сценарий 5. Отклик На Поиск И Confirmed Game

Цель:

- проверить путь от interest/response до понятной подтвержденной игры

Шаги:

1. открытие поиска
2. отклик или approval
3. переход в pending state
4. переход в confirmed game state
5. открытие деталей игры
6. переход в чат или следующий action

Что оценивать:

- единый словарь статусов
- понятность различия между search, response и confirmed game
- согласованность Web и iOS

## Сценарий 6. Chat / Match / Follow-up

Цель:

- проверить, что после подтверждения у пользователя есть ясная следующая точка продолжения

Шаги:

1. открытие мэтча или чата
2. просмотр game request card
3. просмотр статуса
4. follow-up action

Что оценивать:

- не теряется ли пользователь после confirmation
- достаточно ли понятен next step
- нет ли конфликтов между карточкой, чатом и отдельной detail view

## Формат Артефактов

Для каждого сценария команда должна по возможности оставить:

- короткий список экранов и шагов
- screenshots по каждому ключевому шагу
- краткий список проблем по дизайну и UX
- список конкретных файлов, которые нужно менять

## Обязательный Формат Скриншотов

Скриншоты должны сохраняться так, чтобы Product Owner мог сопоставить их со сценарием и шагом.

Рекомендуемая структура:

- `/Users/matvey/Desktop/TennisSearch/.artifacts/design-review/ios/<scenario-id>/step-01-<name>.png`
- `/Users/matvey/Desktop/TennisSearch/.artifacts/design-review/web/<scenario-id>/step-01-<name>.png`

Где:

- `<scenario-id>` — короткий идентификатор сценария, например `01-onboarding`, `02-discover-card`, `03-create-search`
- `<name>` — краткое понятное имя шага, например `welcome`, `sport-level`, `discover-card`, `create-search-form`

## Обязательный Формат Отчета

В отчете по каждому сценарию должно быть:

1. название сценария
2. список шагов
3. screenshot для каждого шага
4. короткое замечание по дизайну на каждом проблемном шаге
5. список рекомендуемых изменений

Если по какому-то шагу screenshot не удалось получить, команда должна явно написать:

- почему именно screenshot отсутствует
- что было сделано вместо него
- остается ли это риском для quality of review

## Минимум Для Ночного Цикла

Если времени не хватает на все сценарии, минимально обязательны:

1. `Первый вход и onboarding`
2. `Discover и player card`
3. `Создание поиска игры`
4. `Отклик на поиск и confirmed game`
