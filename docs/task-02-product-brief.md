## 1) Onboarding — требования (шаги + копирайт)

### 1.1. Определения и инварианты

- **Onboarding** = «быстрый старт», после которого пользователь может открыть подбор игроков (discover) и понять, что дальше делать.
- **Guest onboarding** = сбор черновика профиля без email; email требуется только при защищённом действии.
- **Account onboarding** = сбор минимального профиля в аккаунте; после завершения пользователь попадает в discover в полноценном режиме.
- **Email/верификация**:
  - не запрашиваем “в лоб” на старте;
  - показываем email-шаг **только** когда пользователь пытается выполнить защищённое действие.

**Защищённые действия (триггер email):**
1. “Лайк/интерес” к игроку (создание swipe/интереса).
2. Открытие чата/переписки (если действие требует аккаунта).
3. Сохранение/создание сущности, которая должна пережить устройство: отклик/предложение/игра/поиск, если это запрещено в guest.

### 1.2. Обязательные шаги (must-have) и необязательные (nice-to-have)

#### Шаг 0 (опционально): Intro «как устроен продукт»

Назначение: снять неопределённость “что это за приложение” и “почему не сразу email”.

Копирайт (обязательный смысл, допускается точная формулировка под платформу):
- Заголовок: **«Поиск партнёра без лишних шагов»**
- Подзаголовок: **«Сначала покажем подбор. Профиль соберём быстро. Email спросим только в момент защищённого действия.»**
- 3 буллета-выгоды:
  1) «Подбор по спорту, уровню, району и времени.»
  2) «Два коротких шага: профиль игрока и доступность.»
  3) «Чат, мэтчи и отклики — после email, когда это действительно нужно.»
- CTA: **«Начать»** / **«Собрать профиль игрока»** (вариант зависит от экрана).

Acceptance criteria:
- Экран/блок intro **не должен** блокировать пользователя от старта профиля (есть явный CTA “Начать”).
- Если пользователь уже собрал черновик/профиль, intro не показывается повторно как обязательный шаг.

Edge cases:
- Если хранилище на устройстве недоступно (localStorage/Keychain), intro может показываться повторно, но должен иметь “Понятно/Закрыть” и не ломать навигацию.

#### Шаг 1 (обязательный): База профиля «кто ты»

Минимальный набор полей, без которого **нельзя** открыть подбор:
- Имя: строка, min 2 символа после trim.
- Возраст: минимум 18.
- Город: выбран (даже если фиксированный).

Копирайт:
- Заголовок шага: **«Расскажи о себе»** / **«Профиль игрока»**
- Подзаголовок: **«Только базовая информация, чтобы сразу начать.»**

Acceptance criteria:
- Кнопка “Далее” неактивна, пока имя < 2 или возраст < 18.
- Пользователь видит явное сообщение/подсказку, если город фиксирован (например «Пока запускаемся только в Санкт‑Петербурге.»).

Edge cases:
- Пользователь вводит пробелы вместо имени → считается пустым.
- Пользователь пытается поставить возраст < 18 → UI не позволяет продолжить (и не отправляет данные на сервер).

#### Шаг 2 (обязательный): Спорт/уровень/район + “показывать меня в поиске”

Минимальный набор, без которого **нельзя** открыть подбор:
- Выбран хотя бы 1 вид спорта.
- Для каждого выбранного спорта задан уровень **или** есть явное безопасное значение по умолчанию (если продукт допускает).

Обязательные элементы шага:
- Выбор спорта (multi-select).
- Выбор уровня по каждому выбранному спорту (через guide/подсказку — отдельный sheet допустим).
- Выбор удобных районов (multi-select) с корректным поведением “ничего не выбрано”.
- Переключатель “Показывать меня в поиске / Ищу игру сейчас” (если он остаётся частью модели).

Копирайт:
- Заголовок шага: **«Какой спорт и как тебе удобно играть»**
- Подзаголовок: **«Эти параметры влияют на подбор игроков.»**
- Подсказка по районам: **«Можно выбрать несколько районов. Если не выбирать ничего, покажем игроков шире по городу.»**
- Подсказка к переключателю: **«Ты появишься во вкладке игроков, которые активно ищут партнёра.»**

Acceptance criteria:
- “Далее” неактивна, пока не выбран хотя бы 1 спорт.
- При добавлении спорта система **не теряет** уже выбранные уровни по другим видам спорта.
- Если пользователь снимает спорт, уровень для этого спорта не должен влиять на подбор и не должен отображаться в player card.
- “Ничего не выбрано” по районам:
  - UI объясняет, что подбор будет шире;
  - нет ошибки валидации.

Edge cases:
- Пользователь выбирает много видов спорта: UI обязателен к работе без переполнения (скролл/сетка).
- Пользователь открывает guide по уровням и закрывает его: выбранные значения не сбрасываются.

#### Шаг 3 (необязательный): Доступность «когда удобно»

Назначение: улучшить качество подбора и объяснимость карточек, но не блокировать вход.

Копирайт:
- Заголовок шага: **«Когда тебе удобно»**
- Подзаголовок: **«Этот шаг необязательный. Можно указать позже в профиле.»**
- Инфоблок: **«Укажи дни и интервалы времени, если уже знаешь их. Если нет — заверши онбординг и вернись к этому позже.»**
- CTA:
  - Secondary: **«Пропустить пока»**
  - Primary: **«Начать поиск»** / **«Открыть подбор»**

Acceptance criteria:
- На шаге доступности всегда есть явный “Пропустить пока”, который завершает onboarding и открывает подбор.
- Если пользователь выбрал хотя бы 1 слот, на экране есть компактное резюме выбранного (день + интервал).

Edge cases:
- Пользователь выбирает пересекающиеся интервалы: UI должен приводить к нормализованному виду (без дублей).
- Пользователь удаляет все слоты: резюме должно показывать состояние “Пока не указано / Заполни позже”.

### 1.3. Опыт после onboarding (первый вход в discover)

Acceptance criteria:
- После завершения onboarding пользователь попадает в discover и видит понятный следующий шаг:
  - “смотри игроков” (карточки),
  - “выбери сценарий” (regular/hot), если это часть интерфейса.
- Первый визит в discover может показать обучающий sheet “как устроен поиск”, но он:
  - закрывается явной кнопкой “Понятно”/крестиком,
  - не повторяется после закрытия на этом устройстве (если доступно хранилище).

## 2) Player Card — обязательные блоки и порядок

### 2.1. Зачем player card и какой результат должен дать

Player card должна за 3–5 секунд отвечать на 3 вопроса:
1) “Кто это?”
2) “Почему мне это показывают (почему релевантно)?”
3) “Что я могу сделать дальше (next step)?”

### 2.2. Обязательные блоки (must-have) и порядок сверху вниз

**Блок A — Идентичность и контекст**
1. Аватар.
2. Имя + возраст (если есть).
3. Город + район (если есть).
4. Дистанция (например `distanceLabel`).

**Блок B — Спорт и уровень (короткое резюме)**
5. 1–2 основных спорта с уровнем (например, “Tennis 5”, “Padel 4”).
6. Если спорт/уровней больше: “ещё +N” (или эквивалент), без перегруза.

**Блок C — «Почему подходит» (explainability)**
7. 2–4 причины, отсортированные по полезности для решения “лайк/пропуск”.
   - Приоритет причин:
     1) общий спорт (или совпадение с активным фильтром/поиском),
     2) близкий уровень,
     3) расстояние,
     4) пересечение по доступности (дни/время).
   - Формат: короткие чипы/строки (≤ 40 символов каждая).
   - Примеры формулировок:
     - «Совпадает теннис · уровень рядом»
     - «3 км от тебя»
     - «Играет по вечерам в будни»

Fallback (если данных не хватает):
- показываем минимум 1 причину вида: **«Похожий профиль по спорту и району»** (без упоминания скрытых вычислений).

**Блок D — Предпочтения (вторичный слой)**
8. Формат игры (singles/doubles/both) и поверхность (если релевантно).

**Блок E — Короткое описание**
9. Bio (2–4 строки), с понятным текстом-заглушкой, если bio нет.

**Блок F — Действия (next step)**
10. Primary action: “Можно поиграть” / “Лайк” (на платформах, где это действие есть).
11. Secondary action: “Пропустить”.

### 2.3. Запреты и требования к “score”

Правило:
- Числовой “score” **не должен** быть единственным объяснением релевантности.

Acceptance criteria:
- Если score отображается, он должен быть подписан понятным пользователю смыслом (“Совпадение”, “Релевантность”) и всегда сопровождаться блоком “Почему подходит” (2–4 причины).
- Если причины рассчитаны, но на карточке их нет — это считается багом качества player card.

### 2.4. Edge cases по player card

- Нет района → не показываем “Район: не указан” как главный смысл; показываем “Город” + расстояние, а район как optional.
- Нет bio → показываем нейтральную заглушку без обещаний (“Можно быстро договориться…” допустимо, но без токсичных гарантий).
- Нет доступности → не добавляем “совпало по времени” в причины; используем другие причины.
- У пользователя много спорт‑уровней → карточка не должна распухать: максимум 2 основных + “ещё”.

## 3) Правила modal/sheet vs inline

### 3.1. Термины

- **Inline**: подсказки, ошибки валидации, микро‑пояснения внутри текущего экрана.
- **Toast**: короткое не блокирующее уведомление “получилось/не получилось”.
- **Sheet**: полумодальный слой (bottom sheet), сохраняющий контекст экрана.
- **Modal (blocking)**: полностью блокирующее решение/подтверждение, требующее явного действия.

### 3.2. Decision rules (что выбирать)

Используем **inline**, если:
- пользователь уже в контексте формы/карточки и нужно объяснить поле/ошибку (валидация, подсказка “зачем это нужно”).

Используем **toast**, если:
- нужно подтвердить результат действия без выбора (например “отправлено”, “сохранено”, “ошибка сети”), и контекст менять не нужно.

Используем **sheet**, если:
- нужно дать больше контекста (1–2 экрана) и вернуться назад без потери прогресса;
- нужно провести пользователя через “обучение” (например “как устроен поиск”) без глубокого ухода;
- нужно запросить защищённое действие (email), сохранив точку возврата.

Используем **blocking modal**, если:
- действие необратимо или рискованно и требуется явное подтверждение (например отмена/сброс, если такое есть);
- нужно зафиксировать ключевой выбор, от которого зависит состояние доменной сущности (если без подтверждения высок риск ошибки).

### 3.3. Обязательные применения в рамках Task 02

1) **AuthRequired**:
- всегда sheet (не inline), потому что это смена режима и требуется объяснение “почему”.
- обязательный копирайт‑смысл: “Email нужен, чтобы лайкать/открывать чат/получать ответ; профиль уже собран; вернёмся в тот же контекст после входа.”

2) **Discover intro «как устроен поиск»**:
- sheet, показываем 1 раз на устройство (если возможно).
- закрывается крестиком и кнопкой “Понятно”.
- не должен появляться поверх критического сценария (например поверх формы создания игры).

3) **Гайд уровней спорта (если есть)**:
- sheet, открывается из шага выбора уровня.
- закрытие не должно сбрасывать выбранные уровни.

4) **Match success / “Это мэтч”**:
- допускается sheet или blocking modal (платформенно), но обязательны 2 CTA:
  - “Продолжить поиск”
  - “Открыть чат”

### 3.4. Запреты (чтобы не ломать поток)

- Не открывать sheet/модал для подсказок длиной в 1–2 строки (это inline).
- Не делать цепочки “sheet поверх sheet” в одном сценарии (вместо этого: закрыть текущий → открыть следующий, либо навигация).
- Не переводить пользователя на другой экран “в фоне” при открытом sheet (должен быть понятный момент перехода).
